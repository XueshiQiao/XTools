The notarized app that refused to launch

Four signed, notarized, stapled releases passed every check we could run — and every one was killed by the kernel the instant a user double-clicked it. The culprit was a single restricted entitlement with no provisioning profile to authorize it.

1 · The symptom

A user downloaded the release DMG, mounted it, double-clicked XTools.app — and nothing opened. No window, no Dock bounce, no crash reporter. From the terminal the failure was terse and strange:

# open (LaunchServices path)
$ open /Volumes/XTools/XTools.app
The application cannot be opened for an unexpected reason,
  RBSRequestErrorDomain Code=5 "Launch failed"
  NSPOSIXErrorDomain Code=163 "Launchd job spawn failed"

# exec the binary directly
$ /Volumes/XTools/XTools.app/Contents/MacOS/XTools
Killed: 9        # exit 137 = 128 + SIGKILL. No stdout. No crash log.

Exit 137 is SIGKILL — the process was executed and then killed by the system before it ran a single line of our code, which is why there was no crash report. This is not a bug in the app. It's the launch gate refusing to let the app run.

Figure 1. The kill happens inside amfid during spawn, before main(). The log show --predicate 'process=="amfid"' line names the reason outright — it is looking for a provisioning profile and not finding one.

2 · Why every static check lied

The confusing part: this app is genuinely signed with a Developer ID Application certificate, genuinely notarized by Apple, and the notarization ticket is genuinely stapled. Every tool you'd reach for to "prove" it's fine says it's fine:

Figure 2. The static toolchain validates authenticity; AMFI validates authorization. A signed, notarized DMG can still ship an app that cannot legally run.
Core insight

A green CI pipeline is not proof the app opens. The only proof is to download the artifact, apply a real com.apple.quarantine xattr, and watch it launch. We had four "successful" releases that no downloader could ever open.

3 · Why Debug worked and Release didn't

The tell was that the local Debug build — same entitlements, same Hardened Runtime — launched perfectly, every day. The difference is one file that Xcode quietly adds and manual CI signing does not.

Figure 3. Two ways to be launchable: carry the restricted entitlement with an authorizing profile (Debug), or don't carry it at all (AnyDrag ships zero keychain entitlements). The broken Release did neither.

The root cause is two independent defects that had to both be fixed:

  • Layer 1 — the macro was never expanded. codesign signs the entitlements plist verbatim; unlike Xcode it does not expand $(AppIdentifierPrefix). So the signed keychain group was the literal string $(AppIdentifierPrefix)me.xueshi.xtools.keychain — malformed.
  • Layer 2 — no profile to authorize it. keychain-access-groups is a restricted entitlement (Apple TN3125). Under Hardened Runtime it must be allowlisted by an embedded provisioning profile. Developer ID distribution embeds none by default. Fixing layer 1 alone still died — AMFI wants the profile.

4 · The fix

Keep the entitlement (so both builds use the prompt-free data-protection keychain, staying consistent), and give the release what Xcode gives Debug: an embedded provisioning profile. Concretely, the CI Sign App step now expands the macro and writes a Developer ID profile into the bundle before signing.

# .github/workflows/build.yml — Sign App (essentials)

# 1. Authorize the restricted entitlement: embed the Developer ID profile
#    BEFORE codesign so it's covered by the seal.
echo -n "$MAC_PROVISION_PROFILE_BASE64" | base64 --decode \
  -o "$APP_PATH/Contents/embedded.provisionprofile"

# 2. Expand $(AppIdentifierPrefix) → <TEAMID>. (codesign won't)
sed "s/\$(AppIdentifierPrefix)/${TEAM_ID}./g" "$ENTITLEMENTS" > "$SIGN_ENT"

codesign --force --options runtime --timestamp \
  --entitlements "$SIGN_ENT" \
  --sign "Developer ID Application" "$APP_PATH"

# 3. Guards — fail the build rather than ship the kill again
codesign -d --entitlements - "$APP_PATH" | grep -q "${TEAM_ID}.me.xueshi.xtools.keychain"
test -f "$APP_PATH/Contents/embedded.provisionprofile"

The profile itself is a Developer ID provisioning profile (ProvisionsAllDevices = true), generated from the explicit App ID me.xueshi.xtools and the Developer ID Application certificate. Any explicit App ID auto-includes keychain-access-groups = [TEAMID.*] in its profiles — there is no separate "Keychain Sharing" toggle to hunt for. It's stored as a base64 GitHub secret, never committed.

Why the entitlement at all

The data-protection keychain gates access by entitlement / team, not by the binary's code signature — so it never re-prompts "wants to access the keychain" across rebuilds. The legacy keychain binds its ACL to the app's Designated Requirement, which changes on every re-sign, so it prompts forever. Keeping the entitlement (with a profile) is what buys the prompt-free experience in both Debug and Release.

6 · What would break if you removed each piece

Remove…SymptomSeverity
embedded.provisionprofile embed step Back to square one — AMFI SIGKILLs the notarized app at launch (-413 no matching profile). HIGH
the sed macro expansion Signed keychain group stays the literal $(AppIdentifierPrefix)…; it no longer matches the profile's allowlist → AMFI rejects it. HIGH
the post-sign guards Build stays green while silently shipping the kill again — exactly how four releases went out unnoticed. MEDIUM
the whole entitlement (drop it) App launches fine (matches AnyDrag), but the keychain "wants to access" prompt returns on every signature change — one-time for release users, recurring for debug rebuilds. TRADE-OFF
Reusable rule

Any notarized Developer-ID macOS app that claims keychain-access-groups must embed a provisioning profile to authorize it, or it is SIGKILLed at launch — while every static check passes. This applies to every Developer-ID app in this family that touches the keychain.
