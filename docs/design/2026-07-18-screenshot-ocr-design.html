截图取词（Screenshot OCR）

给 PopBar 增加一个不依赖可选中文本的取词入口：全局快捷键 → 框选屏幕任意区域 → Vision OCR → 自动复制并弹出既有的划词胶囊，所有现有动作原样可用。

决策记录 · Decisions（已锁定）
  1. 触发方式 — 可配置的全局快捷键，默认 ⌘⇧S（Command+Shift+S），可在 PopBar 设置中改键。
  2. 框选方式 — 自绘拖拽选区 overlay + 进程内截屏（不用系统 screencapture）：权限归属清晰，且胶囊能精确锚定到选区。
  3. OCR 语言 — Vision 全语言自动检测（automaticallyDetectsLanguage = true，recognitionLevel = .accurate），不做任何语言配置 UI。
  4. OCR 之后 — 自动复制识别文本到剪贴板 并 弹出胶囊；自动复制是设置开关，默认开。

1 · 背景与目标

PopBar 现有的路径是「划词」：用户用鼠标手势选中文本，胶囊 + 动作轮就在选区旁弹出，跑翻译、摘要、网页预览、复制等动作。它的前提是文本可被选中——而图片里的文字、视频帧、扫描版 PDF、远程桌面、以及各种禁止选择的 App 都不满足这个前提。

截图取词补上这块：按下全局快捷键，在屏幕上任意位置拖出一个矩形，对该区域跑 OCR，把识别出的文本自动复制到剪贴板，并在选区旁弹出同一个胶囊。从胶囊往后的一切与划词完全一致。

能做到「同一个胶囊」，是因为集成缝隙只有一个方法：

PopBarWindowManager.showTransient(
    text: String,        // OCR 识别出的文本
    url: URL?,           // 本路径恒为 nil
    anchor: CGPoint,     // Cocoa 全局坐标（左下原点），胶囊在该点正上方渲染
    actions: [PopBarActionConfig]
)
Core idea

截图取词只是另一种「取词」入口：它最终产出与划词完全相同的 (text, anchor) 二元组。从胶囊往后——动作轮、LLM 动作、复制、网页预览——原样复用，零新概念。

2 · 架构选型

考虑过三种接入方式，取舍如下：

方案思路问题结论
扩展 SelectionStrategy 把 OCR 做成又一种「选区解析策略」 strategy 的契约是从鼠标手势上下文同步解析当前选区；OCR 需要自己的触发器、overlay UI 和异步截屏，塞进去会污染 resolver。 否决
独立新工具 Tab 在侧边栏加一个「截图取词」工具 无法复用胶囊 + 动作轮：要么引入跨工具耦合（违反工具隔离约定），要么把整套胶囊复制一份。 否决
独立前置协调器 PopBar 内部的一个 front-step：自带触发/框选/截屏/OCR，最终通过 showTransient 汇入既有胶囊 现有划词路径零改动；新代码全部收敛在 PopBar/Capture/ 一个子目录里。 采用

采用方案落地为 ScreenOCRController：它持有既有的 PopBarWindowManager 与 ActionStore 引用，串起「快捷键 → 权限 → 框选 → 截屏 → OCR → 剪贴板 + 胶囊」整条链路。PopBarController 只在启用时创建并启动它、停用时注销快捷键——除此之外现有代码不动。

3 · 端到端数据流

Figure 1. 六步主链路 + 三条退出分支。关键时序在 ③→④ 之间：mouse-up 先撤 overlay，等下一个 runloop tick 再截屏，保证遮罩永远不会混进截图。OCR（⑤）整段在后台线程，⑥ 回主线程。

逐步说明（与图中编号对应）：

  1. 触发 — 通过 Carbon RegisterEventHotKey 注册 ⌘⇧S。选 Carbon 而非 NSEvent 全局 monitor 的原因：它是唯一无需 Accessibility 权限且能吞掉按键（不再传给前台 App）的公开全局快捷键 API。
  2. 权限检查 — CGPreflightScreenCaptureAccess() 静默查询 Screen Recording 授权。未授权则调 CGRequestScreenCaptureAccess()（仅首次会弹系统窗）并引导去系统设置，然后终止本次流程——不弹 overlay，避免用户框了半天结果截出黑图。
  3. 框选 — 每块 NSScreen 盖一个无边框透明 overlay 窗口，屏幕整体变暗约 25%，十字光标，拖拽出橡皮筋矩形，实时显示尺寸。Esc / 右键 / 选区小于 8×8 px 都会取消（详见 §4）。
  4. 截屏 — mouse-up 后先撤下全部 overlay，在下一个 runloop tick 用 CGWindowListCreateImage 把选区截成 CGImage（详见 §5）。
  5. OCR — 在后台队列跑 VNRecognizeTextRequest（.accurate + 自动语言检测），按阅读顺序把识别结果拼成一个字符串。主线程全程零阻塞。
  6. 产出 — 回到主线程：结果为空 → 在选区附近弹一个短暂 toast「未识别到文本」即止；非空 → （若自动复制开启）写入 NSPasteboard，然后以选区底边中点为 anchor 调 showTransient 弹出胶囊。

4 · 框选 Overlay 交互

overlay 是本功能唯一的新 UI。目标观感与系统截图（⌘⇧4）一致，用户零学习成本：

Figure 2. 拖拽中的 overlay：遮罩用 even-odd 路径在选区处「挖洞」，选区内保持原始亮度，边框为强调色。尺寸标签跟随拖拽点实时刷新；顶部提示「Esc 取消」常驻。

行为细则

  • 窗口形态 — 每块屏幕一个无边框、透明背景、置顶的 NSWindow（RegionSelectionOverlay），frame 取各自 screen.frame；由 RegionSelectionController 统一创建、协调与撤销。任何一块屏上的操作（取消或完成）会同时撤掉所有屏的 overlay。
  • 光标 — overlay 出现即切换为 crosshair（十字），撤销时恢复。
  • 取消 — Esc、鼠标右键、或松手时选区小于 8×8 px，三者任意其一 → 静默撤下，无任何提示或副作用。
  • 完成 — mouse-up 产出 (选区 rect, 所在 NSScreen)，交给下游截屏。选区 rect 用所在屏幕自己的坐标系记录，再换算成全局坐标（见 Figure 3）。

5 · 截屏与 OCR

截屏：CGWindowListCreateImage

部署目标是 macOS 13.0，所以基线用 CGWindowListCreateImage(rect, .optionOnScreenOnly, kCGNullWindowID, [.bestResolution])。该 API 自 macOS 14 起被标记 deprecated（Apple 推荐 ScreenCaptureKit），但仍然可用；后续可在 macOS 14+ 上条件切换到 SCScreenshotManager，接口收敛在 ScreenCaptureService 一个文件里，升级不外溢。

一个必须小心的坐标细节：CGWindowListCreateImage 接受的是 CG 坐标（原点在主屏左上，y 向下），而 overlay / AppKit 全程用 Cocoa 坐标（原点在主屏左下，y 向上）。换算只需一次翻转：cgY = 主屏高度 − cocoaRect.maxY。这层换算封装在 ScreenCaptureService 内部，调用方只见 Cocoa 坐标。

Figure 3. 多显示器场景：每屏一个 overlay，用各自的 screen.frame；主屏左侧的副屏 origin.x 为负数是常态（本机即如此）。胶囊 anchor 取选区底边中点的 Cocoa 全局坐标，胶囊在其正上方渲染——与划词路径的锚定语义完全一致。

OCR：Vision

// TextRecognizer.swift — 后台队列执行，CGImage 进、String 出
let request = VNRecognizeTextRequest()
request.recognitionLevel = .accurate
request.automaticallyDetectsLanguage = true   // 全语言自动检测，无语言配置 UI

try VNImageRequestHandler(cgImage: image).perform([request])

// 观察结果按阅读顺序拼接：先按 boundingBox 自上而下排序（同一行内自左向右），
// 每个 observation 取 topCandidates(1)，行间以换行连接
let text = sortedObservations
    .compactMap { $0.topCandidates(1).first?.string }
    .joined(separator: "\n")

两点说明：其一，Vision 的 observations 不保证严格阅读顺序，所以显式按 boundingBox 排序而不是依赖返回顺序。其二，整个 handler.perform 在后台队列（.userInitiated）执行，大选区、高分屏也不会卡主线程——识别耗时只影响胶囊出现的延迟，不影响任何交互。

6 · 权限：Screen Recording

整个功能只引入一个新权限：Screen Recording（TCC 面板授权）。仿照现有 AccessibilityAuthorizer 的三段式，新建 ScreenRecordingAuthorizer：

能力API行为
检查 CGPreflightScreenCaptureAccess() 返回 Bool，静默、廉价、不弹窗，可随时调用。
请求 CGRequestScreenCaptureAccess() 触发系统授权弹窗——仅首次有效，之后再调只返回结果不再弹。
去设置 x-apple.systempreferences:
com.apple.preference.security?Privacy_ScreenCapture		 直接打开系统设置的 Screen Recording 面板，用于「去授权」按钮。

PopBar 设置页新增一行权限状态：「屏幕录制：已授权 / 未授权 → 去授权」，样式与现有的 Accessibility 权限行一致。

如实说明的边界

  • 待实测 部分 macOS 版本上，Screen Recording 授权后可能需要重启 App 才生效（Sonoma 及以后通常不需要）——当前系统上的实际行为未验证。兜底策略：授权后首次截屏若返回空图 / 黑帧，提示「已授权，请重启 XTools 生效」。
  • 本路径不需要 Accessibility 权限：Carbon 快捷键、进程内截屏、Vision、弹 NSPanel 都不依赖它。即使 PopBar 的划词监听整个关闭，截图取词也能独立工作。
  • 无需新 entitlement，无需 Info.plist usage string：Screen Recording 是 TCC 面板授权，不走 Info.plist 弹窗描述；Vision 无任何权限要求。（XTools 本就未沙盒化，此处也不需要沙盒例外。）

7 · 文件布局与集成点

全部新代码收敛在 XTools/Sources/Tools/PopBar/Capture/，保持工具自包含：

新文件职责
ScreenRecordingAuthorizer.swiftScreen Recording 权限的检查 / 请求 / 跳系统设置（与 AccessibilityAuthorizer 同构）。
RegionSelectionOverlay.swift单块屏幕的 overlay 窗口 + 橡皮筋拖拽 + 尺寸标签。
RegionSelectionController.swift在所有屏幕铺 overlay，协调取消 / 完成，产出（选区 rect + 所在 NSScreen）。
ScreenCaptureService.swift选区 → CGImage（13+ 用 CGWindowListCreateImage；14+ 可升级 SCScreenshotManager），内含 Cocoa↔CG 坐标翻转。
TextRecognizer.swiftCGImage → 文本，Vision 识别 + 阅读顺序排序拼接。
GlobalHotKey.swiftCarbon RegisterEventHotKey 封装：把 (keyCode, modifiers) 注册为吞键的全局快捷键。
ScreenOCRController.swift总装配：快捷键 → 权限 → 框选 → 截屏 → OCR → 剪贴板 + showTransient。持有既有 PopBarWindowManager 与 ActionStore。

对既有代码的改动仅四处，全部是加法：

  • PopBarController — 启用时创建并启动 ScreenOCRController，停用时注销快捷键。仅此而已。
  • PopBarPreferences — 新增 3 个向后兼容的 key：screenOCREnabled、screenOCRHotKey、screenOCRAutoCopy（默认 true）。老版本配置文件读到未知字段照常忽略并保留。screenOCREnabled 的默认值建议为 false——一个静默吞掉系统级 ⌘⇧S 的默认开关不够保守（见 §8）；此点未在决策中锁定，留待评审拍板。
  • PopBarView — 设置页新增「截图取词」分组：启用开关、快捷键录制器（默认 ⌘⇧S）、自动复制开关、Screen Recording 权限行。
  • Localizable.strings（EN + zh-Hans）— 新增设置项、toast、权限引导等文案，经 L(_:) 读取。

8 · 快捷键与录制器

⌘⇧S 的取舍

⌘⇧S 在很多 App 里是「另存为 / Save As」，而 Carbon 全局快捷键会系统级吞掉这个组合键——注册期间任何 App 都收不到它。默认用 ⌘⇧S 是产品决策（用户明确指定），但必须可改键；设置页的录制器让用户随时换绑。若 RegisterEventHotKey 注册失败（组合键已被其他 App 占用），设置页明确提示「快捷键被占用，请换一个」，而不是静默失效。

关于录制器本身：Naga 工具已有一套快捷键录制 UI（ShortcutRecorder），但 Naga 正在另一个 worktree 里并行开发，此刻抽取共享会制造合并冲突。所以 v1 先做一个 PopBar 本地的小型录制器（录 keyCode + modifiers，能力上是 Naga 版的子集即可），待 Naga 分支合入后再统一抽到共享层——届时是纯重构，无行为变化。

9 · 边界情况与错误处理

场景处理
用户取消（Esc / 右键 / 选区 < 8×8 px） 静默撤下所有 overlay，无提示、无副作用。
OCR 结果为空 选区附近短暂 toast「未识别到文本」；不弹胶囊、不写剪贴板。
截屏返回 nil / 黑帧（权限被中途吊销等） toast 提示 + 引导到 Screen Recording 设置面板。
授权后首次截屏仍是黑帧 提示「已授权，请重启 XTools 生效」（是否必要待实测，见 §6）。
多显示器，含主屏左侧 origin.x 为负的屏 每屏一个 overlay、各用各的 screen.frame；选区坐标按所在屏换算（Figure 3）。
overlay 混入截图 结构性规避：mouse-up 先撤 overlay，下一个 runloop tick 再截屏。
快捷键被其他 App 占用 RegisterEventHotKey 返回失败 → 设置页显示「快捷键被占用，请换一个」。
超大选区 / 高分屏 Vision 可直接处理；OCR 在后台线程，主线程零阻塞，只体现为胶囊出现稍慢。

10 · 隐私与数据安全

  • 截图不落盘 — 截取的 CGImage 只在内存中存在，OCR 完成即释放；与系统 screencapture 会写文件不同，这里永不写盘。
  • 剪贴板只放文本 — 自动复制写入的是识别出的文本，不是图片。
  • 日志零内容 — 日志只记录长度、耗时、成功与否，绝不记录识别出的文本内容（屏幕上可能是密码、聊天记录等敏感信息）。

11 · v1 范围裁剪（YAGNI）

  • 只做矩形框选——不做窗口拾取、全屏截取、自由形状。
  • 不做语言配置 UI——全语言自动检测已覆盖需求。
  • 不做截图历史 / OCR 历史。
  • 识别文本直接进胶囊——不加「先编辑再执行动作」的中间步骤。
