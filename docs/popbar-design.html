PopBar — 划词胶囊工具条

选中文本即弹出一条可定制的 AI 动作胶囊。本文剖析 Easydict / Xpop / SelectedTextKit 三个参考实现的取词内核,并据此给出 XTools 的四层、可插拔架构 —— 取词层被抽象成一个有序回退的策略接口,今天只装一种,明天能塞五种。

1 · 三个参考实现教会我们的事

这套功能的难点不在 UI,而在“可靠地拿到选中文本”。三个开源项目把这条路趟出了血:它们的回退链高度一致 —— Accessibility 优先 → WebKit 走 TextMarker → 兜底模拟复制读剪贴板 —— 差异只在“谁来编排回退”和“代码成熟度”。

项目取词机制触发定位成熟度 / 取舍
Easydict
production		 AX → AppleScript(浏览器)→ 模拟 Cmd+C / 菜单 Copy。按 App + AXError 的强制复制白名单(Word/VSCode/Firefox/ChatGPT…) NSEvent 全局监听 + 手势推断;CGEventTap 仅在胶囊可见时捕获按键以消解 鼠标点(kAXBoundsForRange 实现了但注释掉) 久经沙场。值得抄:回退编排、按 App 容错表、菜单项多语言匹配、changeCount 轮询、消除提示音。坑:回退策略散落在库与 App 两侧
SelectedTextKit
SPM 库		 同上五策略,封装成 TextStrategy 枚举 + 大 switch —(只管取词) getSelectedTextFrame() 暴露选区矩形 取词内核可直接 SPM 引入。但抽象是“枚举 + switch”而非协议,.auto 与有序数组两套回退重复 —— 是我们要避免的反面教材

三者的共识

取词永远是有序回退,不是单一方案;成功判据是“非空字符串”;唯一应中断整条链的错误是辅助功能权限被拒。我们把这条共识固化进一个 SelectionResolver,且只此一处编排回退 —— 这正是三个参考实现都没做好的地方。

2 · 四层架构与数据流

整条管线拆成四个互不知道彼此内部的层,接缝处只传值类型(SelectionResult / CGPoint / 回调闭包)。触发层不认识窗口,取词层不认识按钮。

Figure 1. 四层各自可替换:换触发手势不动取词,换取词策略不动窗口。协调器是唯一知道全流程的对象,且严格主线程。

3 · 取词层:你要的那层抽象

这是本设计的核心诉求。我们不把取词写死成某一种方案,也不直接依赖 SelectedTextKit;而是定义一个策略协议,每种取词方式是一个独立实现,由 SelectionResolver 按顺序尝试、首个非空者胜出。新增一种兜底 = 新写一个 SelectionStrategy 并在 resolver 注册,管线其余部分一行不改。

// 传入“在哪取、给谁取”,任何策略都不直接摸全局状态 —— 可测、可组合。
struct SelectionContext { let frontmostApp: NSRunningApplication?; let mouseLocation: CGPoint }

// 成功取到的结果。via 记录“是哪种策略赢的”(埋点 + 未来按来源做差异处理)。
struct SelectionResult { let text: String; let via: SelectionStrategyID; var bounds: CGRect? }

// 一种取词方式。加新策略只需实现它 —— 这就是可插拔、可融合的接缝。
protocol SelectionStrategy: AnyObject {
    var id: SelectionStrategyID { get }
    func canHandle(_ ctx: SelectionContext) -> Bool          // 便宜的预筛(如 AppleScript 仅限受支持浏览器)
    func selectedText(_ ctx: SelectionContext) async throws -> SelectionResult?
}

编排只此一处。空字符串视为失败、继续下一个;.permissionDenied 是唯一会立刻中断整链的致命错误 —— 这正是三个参考实现的共识,被我们收进一个对象:

func resolve(_ ctx: SelectionContext) async -> SelectionResult? {
    for s in strategies where s.canHandle(ctx) {
        do { if let r = try await s.selectedText(ctx), !r.text.isEmpty { return r } }
        catch let e as SelectionError where e.isFatal { return nil }  // 权限被拒,放弃
        catch { continue }                                          // 该策略失败,试下一个
    }
    return nil
}
Figure 2. v1 装两节:AX 快路 + 剪贴板兜底。虚线节点是预留槽 —— 加策略不改这张图的形状,只往数组里追加。

v1 的两种策略

  • AccessibilityStrategy — systemWideElement → 焦点元素 → kAXSelectedTextAttribute;失败再试 WebKit 的 AXSelectedTextMarkerRange + AXStringForTextMarkerRange。对 Electron/Chromium 取词为空时,一次性给目标 App 设 AXManualAccessibility/AXEnhancedUserInterface 打开其 AX 树。顺带用 kAXBoundsForRange 取选区矩形(拿得到就用,作为更精准定位)。
  • ClipboardCopyStrategy — 备份整块剪贴板(所有 type)→ 用 CGEvent 合成 Cmd+C → 5ms 轮询 changeCount(上限 ~400ms)确认落地 → 读字符串 → 还原剪贴板。非破坏性,且不依赖任何额外权限/entitlement。

4 · 触发、窗口、动作层

触发层 —— 为什么不用 CGEventTap

用 NSEvent.addGlobalMonitorForEvents 监听鼠标按下/拖拽/抬起 + 双击,广播给一组手势识别器(DragSelectGesture:down 与 up 之间拖拽事件数 ≥ 阈值;DoubleClickGesture:clickCount ≥ 2)。全局监听只观察、不吞事件 —— 正合我们意图。刻意避开 CGEventTap:它存在“代码签名 + 静默禁用竞态”,系统会悄悄关掉它,可靠性差。手势触发后取鼠标抬起点(全局监听里 locationInWindow 即屏幕坐标)。

窗口层 —— 不抢焦点的胶囊

三个项目的 NSPanel 配方一致,直接采用 —— 关键是 .nonactivatingPanel,胶囊弹出时不把焦点从源 App 抢走(否则源选区即丢):

window = NSPanel(contentRect: rect,
                 styleMask: [.fullSizeContentView, .nonactivatingPanel],
                 backing: .buffered, defer: false)
window.level = .floating
window.collectionBehavior = [.canJoinAllSpaces, .ignoresCycle]
window.isOpaque = false; window.backgroundColor = .clear      // SwiftUI 圆角胶囊 + 毛玻璃
window.orderFront(nil)                                       // 永不 makeKey —— 保住源 App 焦点

内容是 NSHostingView 托管的 SwiftUI 胶囊。定位:居中于鼠标点上方,并按 screen.visibleFrame 贴边裁剪(补上 Xpop 缺失的越界保护)。胶囊外的任何点击/滚动/按键由一个临时全局监听消解。

动作层 —— v1 用假数据

动作是数据驱动的:复制(真实,写剪贴板)、翻译/润色/解释(FakeAIService 返回占位文本)。点 AI 动作后,胶囊原地切到结果视图展示假输出 + 一个“复制结果”按钮 —— 完整跑通“选词→动作→结果”闭环,接真模型时只换 FakeAIService 一处。按钮全区域可点(.contentShape(Rectangle()))。

5 · 权限、验证与边界

v1 只需辅助功能(Accessibility)权限(运行时 AXIsProcessTrusted 检测 + 引导授权),复用 XTools 既有的非沙盒形态,不新增任何 entitlement 或 Info.plist 用途串。功能 opt-in:授权并在设置页打开开关后才启动监听。设置页提供一个“预览胶囊”按钮(亦受 XTOOLS_POPBAR_PREVIEW=1 环境变量触发),让胶囊在固定位置以样例文本显示 —— 既是给用户的预览,也是我无需真机划词即可截图验证 UI 的路径。

诚实边界

我能亲自验证:编译通过、应用启动、PopBar 出现在侧栏、胶囊在预览路径下正确渲染、监听启动(看日志)。我无法在不手动授予辅助功能权限并真机划词的情况下,验证“任意 App 里全局取词”的端到端效果 —— 那一步标注为未验证,待你授权后实测。代码已按“一次复现即定位”的原则布满 DBG 日志。
